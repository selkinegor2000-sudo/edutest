# Материал для актуализации теоретической части диплома

Ниже приведен переработанный текст практико-ориентированных разделов дипломной работы. Текст согласован с фактической реализацией проекта EduTest: веб-приложение построено на клиент-серверной архитектуре, использует React, TypeScript, Express, SQLite, Drizzle ORM и API языковой модели для интеллектуальной оценки развернутых ответов.

## 2.2. Разработка логической и физической структуры базы данных для хранения учебных материалов и результатов тестирования

Ключевым этапом проектирования разработанной системы стало формирование структуры данных, обеспечивающей хранение учебных материалов, тестов, ответов студентов и результатов интеллектуального анализа. Поскольку разрабатываемое приложение ориентировано на использование в учебном процессе и должно поддерживать несколько ролей пользователей, логическая модель базы данных строилась вокруг сущностей, отражающих основные объекты предметной области: преподавателей, студентов, тестов, вопросов, вариантов ответов, попыток прохождения и результатов оценивания.

В качестве основной сущности выступает пользователь системы. В рамках проекта предусмотрены две роли: преподаватель и студент. Для каждой учетной записи сохраняются идентификатор, логин, хеш пароля, полное имя, роль и дополнительные персональные сведения, используемые для настройки профиля. Такое решение позволяет использовать единую модель аутентификации и авторизации, а различие в доступных действиях определяется бизнес-логикой приложения.

Следующей важной сущностью является тест. Для каждого теста в системе сохраняются его название, описание, учебная дисциплина, идентификатор преподавателя-автора, временное ограничение, признаки публикации и соревновательного режима, а также временные метки создания и обновления. Такая структура обеспечивает не только хранение содержания теста, но и поддержку сценариев повторного использования материалов, публикации готовых заданий для студентов и формирования рейтингов для конкурсных режимов проверки знаний.

С тестом логически связана сущность вопроса. Каждый вопрос принадлежит конкретному тесту и характеризуется типом, формулировкой, количеством баллов, эталонным ответом и порядком отображения. В разработанном прототипе поддерживаются вопросы с одним правильным вариантом, вопросы с несколькими правильными вариантами, а также вопросы с развернутым открытым ответом. Для мультимедийного расширения предусмотрены поля для хранения ссылок на изображение или видеоматериал, что делает модель пригодной для дальнейшего развития системы и ее адаптации к различным дисциплинам.

Для вопросов закрытого типа используется отдельная сущность вариантов ответа. Каждый вариант связан с вопросом по внешнему ключу и содержит текст варианта, признак правильности и порядковый номер. Такое разделение позволяет гибко хранить любое количество вариантов и независимо управлять их содержанием, не перегружая таблицу вопросов. Кроме того, подобная структура облегчает генерацию интерфейса тестирования на клиентской части и упрощает изменение содержания теста преподавателем.

Центральной сущностью, отражающей процесс контроля знаний, является попытка прохождения теста. В ней фиксируются идентификатор теста, идентификатор студента, статус попытки, время начала и завершения, итоговый балл, максимальное количество баллов и сохраненные ответы. Наличие статуса позволяет различать незавершенные, завершенные и просроченные попытки. Это важно для обеспечения устойчивой логики работы системы: студент может продолжить прерванный тест, а преподаватель получает корректную картину активности обучающихся.

С попыткой связана таблица отдельных ответов. Для каждого ответа сохраняются идентификаторы попытки и вопроса, текст ответа, выбранные варианты, признак корректности, начисленные баллы, а также результаты интеллектуального анализа. В частности, в базе данных предусмотрены поля для хранения AI-оценки и AI-комментария. Это решение принципиально важно для рассматриваемой темы дипломной работы, поскольку позволяет сохранять не только итоговый числовой результат, но и интерпретацию ответа, сформированную языковой моделью.

Для поддержки учебного сценария, в котором преподаватель работает с закрепленной группой студентов, введена дополнительная сущность связи преподавателя и студента. Она реализует отношение многие-ко-многим и позволяет закреплять обучающихся за конкретным преподавателем. Такое проектное решение обеспечивает основу для персонализированного контроля, формирования групповой аналитики и разграничения доступа к данным в зависимости от роли пользователя.

При реализации физической структуры базы данных в качестве СУБД была выбрана SQLite. Данное решение обусловлено несколькими факторами. Во-первых, система разрабатывалась как прототип, для которого важны простота развертывания и отсутствие сложной инфраструктурной зависимости от отдельного сервера баз данных. Во-вторых, SQLite хорошо подходит для учебных и демонстрационных проектов, так как база данных хранится в одном файле и может быть легко перенесена между рабочими средами. В-третьих, выбранный вариант обеспечивает достаточную производительность для сценариев локальной или малой групповой эксплуатации.

Для доступа к данным используется ORM Drizzle, которая обеспечивает строго типизированное описание схемы и уменьшает риск ошибок при работе с таблицами и связями. Схема данных описана в коде приложения, что позволяет поддерживать согласованность между серверной логикой, типами клиентской части и процедурами миграции. В результате достигается единое информационное пространство проекта: структура данных, используемая сервером, одновременно выступает источником типов для клиентского приложения.

Особое внимание при проектировании физической структуры было уделено ограничениям целостности. В таблицах реализованы первичные и внешние ключи, каскадное удаление для зависимых сущностей и уникальность логинов пользователей. За счет этого обеспечивается непротиворечивость данных при удалении тестов, вопросов и попыток. Например, при удалении теста связанные вопросы и варианты ответов не остаются в базе в виде «висячих» записей, что положительно влияет на надежность системы и корректность аналитики.

Таким образом, разработанная логическая и физическая структура базы данных отражает реальные процессы контроля знаний в образовательной среде и обеспечивает хранение как традиционных результатов тестирования, так и данных интеллектуального анализа. Это создает основу для последующей аналитики, построения отчетности и адаптивной поддержки преподавателя и студента.

## 2.3. Проектирование пользовательского интерфейса и взаимодействия с системой

Пользовательский интерфейс системы проектировался исходя из необходимости обеспечить простой и понятный доступ к функциям контроля знаний для двух категорий пользователей: преподавателей и студентов. При разработке учитывались требования интуитивности, адаптивности, минимизации когнитивной нагрузки и наглядного представления результатов. С учетом этих требований была выбрана модель одностраничного веб-приложения, в которой навигация между разделами выполняется без полной перезагрузки страницы.

Интерфейс преподавателя ориентирован на управление образовательным контентом и анализ результатов. В системе преподаватель может создавать тесты, редактировать вопросы, указывать правильные ответы, задавать временные ограничения, публиковать тесты и отслеживать статистику прохождения. Особое значение имеет возможность просматривать результаты по каждому тесту и по каждому студенту, так как именно на этом уровне преподаватель принимает решения о корректировке учебного процесса. В интерфейс были включены элементы визуальной аналитики, позволяющие оценить средний результат группы, число попыток и особенности выполнения заданий.

Интерфейс студента, напротив, строится вокруг сценария прохождения теста и анализа собственного результата. В личном кабинете обучающийся видит список доступных тестов, сведения о ранее пройденных работах, свой процент выполнения и информацию о завершенных попытках. Если тест уже был сдан, система отображает соответствующий статус, а при высоком результате выделяет достижение визуальным маркером. Такой подход выполняет не только информирующую, но и мотивационную функцию.

Особое внимание уделялось экрану прохождения теста. В рамках проекта реализован пошаговый сценарий выполнения заданий с отображением номера текущего вопроса, общего числа вопросов, индикатора прогресса и навигации между вопросами. Наличие визуальных маркеров для уже отвеченных пунктов позволяет студенту лучше контролировать собственное продвижение по тесту и снижает вероятность пропуска заданий. С точки зрения педагогического дизайна это повышает прозрачность процедуры контроля и уменьшает стрессовую нагрузку во время прохождения проверки.

После завершения теста студент получает страницу результатов, где отображаются итоговый процент, набранные баллы, количество правильных и неправильных ответов, время выполнения и подробный разбор каждого ответа. Для вопросов открытого типа система показывает AI-комментарий, сформированный на основе сравнения ответа студента с эталонным вариантом. Дополнительно реализован блок сравнительного анализа, в котором собственный результат сопоставляется со средним результатом по группе. Этот элемент особенно важен с точки зрения обратной связи, поскольку помогает студенту оценить свой уровень не изолированно, а в контексте общей успеваемости.

С технической точки зрения клиентская часть разработана на React с использованием TypeScript, что обеспечивает модульность интерфейса и повышает надежность разработки. Для управления серверными запросами применяется TanStack Query, позволяющий централизованно работать с загрузкой данных, кэшированием и актуализацией интерфейса. Набор UI-компонентов строится на базе shadcn/ui и Tailwind CSS, что позволило сформировать единый визуальный стиль приложения и ускорить реализацию типовых интерфейсных элементов.

Интерфейс адаптирован для работы на различных размерах экрана. Это важно в условиях современной образовательной практики, где значительная часть пользователей взаимодействует с учебными системами не только с настольных компьютеров, но и с ноутбуков и мобильных устройств. Адаптивность достигается за счет гибкой сетки компонентов, перераспределения блоков и использования компактных вариантов карточек и навигации.

Таким образом, интерфейс системы не является лишь оболочкой над алгоритмами контроля знаний, а выступает полноценным средством организации образовательного взаимодействия. Он обеспечивает преподавателю инструменты управления и анализа, а студенту — понятный сценарий прохождения тестирования и получения содержательной обратной связи.

## 2.4. Выбор стека технологий и среды разработки для прототипа

При выборе технологического стека для реализации прототипа учитывались требования к скорости разработки, простоте сопровождения, доступности библиотек, переносимости и удобству локального развертывания. Разрабатываемая система должна была объединять веб-интерфейс, серверную бизнес-логику, хранение данных и интеграцию с внешним AI-сервисом. По этой причине предпочтение было отдано современному стеку JavaScript/TypeScript, который позволяет использовать единый язык программирования как на клиентской, так и на серверной стороне.

Клиентская часть реализована на React. Выбор данного фреймворка обусловлен его компонентной архитектурой, высокой распространенностью и развитой экосистемой. React позволяет удобно декомпозировать пользовательский интерфейс на независимые элементы: страницы, карточки тестов, формы, таблицы статистики, блоки аналитики и навигационные элементы. Использование TypeScript повышает типовую безопасность и особенно важно в проекте, где данные пользователя, тестов и результатов активно передаются между клиентом и сервером.

В роли сборщика и среды фронтенд-разработки используется Vite. Его применение позволяет сократить время запуска проекта и ускорить обновление интерфейса в процессе разработки. Для стилизации интерфейса выбран Tailwind CSS, обеспечивающий гибкое построение адаптивных макетов и единообразное оформление компонентов. Дополнительные пользовательские элементы и примитивы интерфейса построены на базе набора shadcn/ui, что позволило быстро реализовать карточки, таблицы, бейджи, формы и модальные окна.

Серверная часть приложения реализована на Express с использованием TypeScript. Express был выбран как достаточно легковесный и хорошо документированный фреймворк, пригодный для построения REST API. На сервере реализованы функции аутентификации пользователей, маршруты для управления тестами, логика запуска и завершения попыток, сохранение ответов, получение статистики и взаимодействие с AI-модулем. Для управления сессиями используется express-session в сочетании с memorystore, что обеспечивает поддержку авторизованных пользовательских сессий без избыточного усложнения инфраструктуры.

Для безопасной работы с учетными данными пользователей применяется библиотека bcrypt, с помощью которой пароли сохраняются в виде хешей. Такой подход соответствует базовым требованиям информационной безопасности и предотвращает хранение паролей в открытом виде в таблицах пользователей. Валидация пользовательских данных на клиенте и сервере частично унифицирована благодаря использованию Zod и drizzle-zod.

В качестве системы хранения данных, как отмечалось выше, была выбрана SQLite. Для взаимодействия с ней используется better-sqlite3 и ORM Drizzle. Такой стек позволил получить строго типизированный доступ к данным, удобную схему миграций и высокую скорость работы в локальном прототипе. Отказ от отдельного серверного решения для базы данных существенно упростил запуск проекта в условиях учебной работы и на компьютере разработчика.

Отдельное место в выбранном стеке занимает интеграция с внешним AI-сервисом. Для анализа развернутых ответов используется обращение к API Groq с моделью Llama 3.3 70B Versatile. Такой подход позволил отказаться от локального развертывания тяжелой модели и при этом получить качественный семантический анализ текстовых ответов. С точки зрения проектирования это рациональное решение, поскольку оно уменьшает требования к вычислительным ресурсам клиентского устройства и переносит наиболее сложную обработку на внешнюю сервисную инфраструктуру.

Итоговый стек технологий оказался сбалансированным по следующим критериям: скорость разработки, понятность архитектуры, возможность быстрого тестирования прототипа, удобство локального запуска и достаточная функциональность для интеграции AI-компонента. Благодаря этому была получена технологическая основа, пригодная как для демонстрации прототипа, так и для его последующего расширения.

## 2.5. Разработка модуля обработки естественного языка для анализа ответов

Одной из ключевых особенностей разработанной системы является наличие интеллектуального модуля оценки открытых ответов студентов. В отличие от заданий закрытого типа, где корректность может быть определена прямым сравнением выбранных вариантов с эталоном, задания с развернутым ответом требуют учета смыслового содержания текста, полноты раскрытия темы и степени соответствия ожидаемому знанию. Именно поэтому в проекте был реализован модуль интеллектуального анализа на основе внешней языковой модели.

Разработанный модуль работает по схеме прикладного промпт-инжиниринга. Для каждого открытого вопроса в систему передаются три элемента: формулировка вопроса, эталонный ответ преподавателя и текст, введенный студентом. На основании этих данных формируется структурированный запрос к языковой модели. В запросе задается шкала оценивания от 0 до 100 баллов и явное требование вернуть результат в формате JSON, содержащий числовую оценку и текстовый комментарий. Такой подход обеспечивает формальную определенность ответа AI-модуля и облегчает его дальнейшую обработку на сервере.

Содержательно работа модуля строится не на поиске полного лексического совпадения, а на приближенной семантической оценке. Модель анализирует, насколько ответ студента соответствует ожидаемому смыслу, есть ли в нем ключевые понятия, отсутствуют ли принципиальные ошибки и насколько полно раскрыта тема. В результате студент получает не только оценку, но и объяснение, где указываются сильные стороны ответа и направления для улучшения. Тем самым система решает сразу две задачи: контроль знаний и формирующую обратную связь.

С точки зрения архитектуры AI-модуль встроен в серверную часть. При проверке открытого ответа сервер вызывает функцию интеллектуальной оценки, которая обращается к API модели через HTTPS-запрос. После получения результата сервер проводит базовую валидацию: ограничивает значение оценки допустимым диапазоном и сохраняет комментарий модели в базе данных. Это позволяет в дальнейшем отобразить AI-анализ в интерфейсе результатов и использовать его при разборе выполненной работы.

Важным аспектом разработки стала обработка отказоустойчивости. Если внешний AI-сервис временно недоступен, не настроен ключ доступа или возвращает некорректный ответ, система не прекращает работу. Вместо этого используется безопасный резервный сценарий: студент получает сообщение о временной недоступности AI-анализа, а ответ может быть дополнительно проверен преподавателем. Такой механизм особенно важен для образовательных систем, поскольку исключает полную блокировку процесса контроля при временных сетевых или сервисных сбоях.

Следует отметить, что в данной работе AI-модуль не обучается на собственном локальном корпусе ответов студентов, а использует готовую большую языковую модель как внешний интеллектуальный сервис. Для задач прототипирования это является оправданным решением. Во-первых, оно позволяет быстро получить работоспособный результат без создания большой размеченной выборки. Во-вторых, использование мощной языковой модели обеспечивает достаточно высокое качество анализа ответов даже при вариативности формулировок. В-третьих, такой подход хорошо демонстрирует перспективность интеграции современных генеративных моделей в образовательные системы.

Таким образом, разработанный модуль обработки естественного языка реализует интеллектуальный анализ открытых ответов на практическом уровне и является важнейшим элементом разработанной системы контроля знаний. Именно он отличает создаваемое приложение от традиционных тестирующих систем, ограниченных лишь проверкой по заранее заданным вариантам.

## 2.6. Интеграция ИИ-модуля в общую архитектуру системы и создание прототипа

После разработки основных компонентов системы был реализован их единый сценарий взаимодействия в рамках клиент-серверной архитектуры. Прототип построен таким образом, чтобы пользовательский интерфейс, серверная логика, база данных и AI-модуль взаимодействовали как части единой информационной среды. Архитектурно система разделена на три уровня: клиентский уровень, сервер приложений и уровень данных с внешним AI-сервисом.

Клиентский уровень отвечает за взаимодействие с конечным пользователем. Преподаватель через веб-интерфейс создает тест, формирует вопросы и публикует материал, а студент авторизуется, выбирает доступный тест и приступает к выполнению. На этом уровне интерфейс обеспечивает ввод данных, навигацию между вопросами, отображение прогресса, получение результатов и визуализацию аналитики.

Серверный уровень выступает центральным координирующим звеном системы. Все действия пользователя оформляются в виде HTTP-запросов к REST API. Сервер принимает запросы, выполняет проверку авторизации и прав доступа, обрабатывает бизнес-логику и обращается к базе данных. Например, при запуске теста сервер создает новую попытку прохождения, но одновременно предотвращает повторное прохождение уже завершенного теста. Это решение реализует важное организационное требование объективности контроля знаний.

При отправке ответов студентов сервер выполняет различную логику в зависимости от типа вопроса. Для заданий с выбором ответа используются детерминированные правила проверки, основанные на сравнении выбранных вариантов с эталонной конфигурацией. Для открытых ответов запускается описанный выше модуль интеллектуальной оценки. Возвращенные моделью балл и комментарий сохраняются в таблице ответов, после чего используются при расчете итогового результата попытки.

Уровень данных реализован на основе SQLite, а взаимодействие с ним происходит через Drizzle ORM. Такое решение обеспечивает изоляцию логики хранения от маршрутов API и упрощает сопровождение проекта. Отдельный слой хранения реализует операции чтения и записи пользователей, тестов, попыток, статистики, связей преподавателя и студентов. Благодаря этому серверные маршруты остаются относительно компактными и сосредоточены на прикладной логике, а не на деталях низкоуровневой работы с SQL.

Интеграция внешнего AI-сервиса встроена в процесс обработки ответа как специализированный серверный вызов. Это архитектурно оправдано по двум причинам. Во-первых, API-ключ не передается на клиентскую сторону, что повышает безопасность. Во-вторых, сервер получает возможность централизованно управлять форматами запросов, обработкой ошибок и логикой сохранения результатов. Таким образом, AI-модуль становится не внешним дополнением, а органической частью архитектуры приложения.

В рамках создания прототипа была реализована не только минимальная функциональность, но и ряд практических механизмов, повышающих качество пользовательского опыта. К ним относятся сохранение прогресса попытки, разбор ответов после завершения теста, сравнение результата студента со средним результатом по группе, соревновательный режим с рейтингом, а также экспорт данных. В совокупности это позволило получить прототип, близкий к прикладному образовательному сервису, а не просто демонстрационный набор отдельных функций.

Следовательно, интеграция ИИ-модуля в общую архитектуру была реализована успешно: интеллектуальная оценка встроена в пользовательский сценарий, результаты сохраняются и используются в аналитике, а сама система функционирует как единый программный комплекс.

## 2.7. Разработка тестовых сценариев и сбор модельных данных

Для апробации разработанного прототипа потребовалось сформировать набор модельных данных и подготовить тестовые сценарии, имитирующие типичные действия преподавателей и студентов. Такой подход позволил проверить корректность работы системы в условиях, приближенных к реальному образовательному процессу.

На этапе наполнения базы данных были созданы тестовые учетные записи преподавателей и студентов, а также связи между ними. Это позволило воспроизвести структуру небольшой учебной группы, в которой преподаватели публикуют тесты, а студенты проходят контрольные мероприятия и получают результаты. Кроме того, были сформированы тесты по нескольким дисциплинам, включающие вопросы разных типов: с одним правильным вариантом, с несколькими правильными вариантами и с развернутым ответом.

Тестовые сценарии разрабатывались так, чтобы охватить основные функциональные ветви приложения. В первую группу вошли сценарии регистрации, аутентификации и разграничения доступа. Проверялась корректность входа в систему, назначение ролей, защита маршрутов преподавателя и недоступность административных функций для студентов. Данный класс сценариев особенно важен, поскольку ошибка в разграничении доступа способна привести к искажению образовательного процесса и нарушению конфиденциальности данных.

Во вторую группу вошли сценарии создания и публикации тестов. Проверялась возможность преподавателя сформировать тест, добавить вопросы, сохранить варианты ответов, задать время прохождения и опубликовать материал. Дополнительно оценивалась корректность отображения созданного теста в списке доступных для студентов. Такой набор сценариев позволил подтвердить работоспособность полного цикла подготовки контрольного мероприятия.

Третью группу составили сценарии прохождения тестов студентами. Проверялись запуск попытки, переход между вопросами, сохранение ответов, завершение теста, расчет результата и запрет повторного прохождения уже выполненной работы. Наличие запрета на повторное завершенное прохождение представляет собой важную организационную меру, обеспечивающую корректность учета результатов и честность контрольной процедуры.

Отдельный блок сценариев был посвящен открытым ответам и AI-анализу. Для таких сценариев проверялись корректность передачи формулировки вопроса, эталонного ответа и текста студента в AI-модуль, получение числовой оценки, сохранение комментария модели и отображение результата в пользовательском интерфейсе. Дополнительно анализировалось поведение системы в случае отсутствия ответа от внешнего сервиса. Такой подход позволил убедиться, что даже при частичной недоступности AI-компонента система не теряет общую работоспособность.

Наконец, были сформированы аналитические сценарии: просмотр результатов теста преподавателем, получение агрегированной статистики по тесту, отображение сравнения результата студента со средним значением по группе, а также работа рейтинга в соревновательных тестах. Эти сценарии особенно важны для подтверждения прикладной ценности системы, поскольку показывают, что разработанный прототип поддерживает не только сам факт оценивания, но и последующий аналитический этап.

Таким образом, сбор модельных данных и проектирование тестовых сценариев обеспечили содержательную проверку разработанной платформы и позволили перейти от уровня изолированной разработки модулей к оценке целостного поведения системы в реальных пользовательских цепочках.

Проверка работоспособности разработанного прототипа проводилась поэтапно и включала модульное, интеграционное и пользовательское тестирование. На модульном уровне оценивалась корректность отдельных компонентов: маршрутов авторизации, операций работы с тестами, функций хранения данных, механизмов расчета результата и процедур AI-анализа. Интеграционное тестирование было направлено на проверку совместной работы клиентской и серверной частей, а также на корректность взаимодействия приложения с базой данных и внешним AI-сервисом.

В ходе тестирования было подтверждено, что система корректно реализует жизненный цикл работы с тестом: от его создания преподавателем до получения студентом детального разбора результата. Пользователь может войти в систему в соответствии со своей ролью, получить доступ только к разрешенным функциям, выполнить тест и получить итоговый результат с аналитической интерпретацией. Это свидетельствует о согласованности логики маршрутов, модели данных и клиентского интерфейса.

В каждом из этих случаев система демонстрировала корректное поведение: отклоняла несанкционированные действия, возвращала информативные сообщения об ошибке и сохраняла общую работоспособность. Анализ полученных результатов показал, что наиболее ценной особенностью системы является сочетание автоматической количественной оценки и содержательной текстовой обратной связи. В отличие от традиционных тестирующих систем, в которых студент получает только итоговый балл или список ошибок, разработанный прототип способен предоставлять комментарий к открытому ответу. Это повышает педагогическую значимость системы, так как переводит контроль знаний из сугубо фиксирующей процедуры в инструмент сопровождения обучения.

Однако эти ограничения не снижают ценность полученного результата, а, напротив, указывают направления дальнейшего развития системы.

В целом результаты тестирования показывают, что созданный прототип может рассматриваться как работоспособная основа для дальнейшего расширения. ## 2.9. Оценка точности, надежности и удобства использования разработанной системы

Оценка эффективности разработанной системы проводилась по трем основным направлениям: точность контроля знаний, надежность функционирования и удобство использования для конечных пользователей. Такой подход обусловлен тем, что образовательная система не может считаться качественной, если она показывает хорошие алгоритмические результаты, но неудобна в эксплуатации или нестабильна в работе.

Точность системы проявляется в корректности определения результата прохождения теста и в адекватности оценки открытых ответов. Для вопросов закрытого типа точность обеспечивается детерминированной логикой сравнения правильных вариантов и выбранных ответов, что дает воспроизводимый и однозначный результат. Для открытых ответов точность зависит от согласованности эталонного ответа преподавателя, качества промпта и возможностей используемой языковой модели. Хотя такая оценка носит вероятностный характер, в рамках прототипа она демонстрирует приемлемый уровень содержательной интерпретации и позволяет формировать полезную обратную связь.

Надежность системы определяется ее способностью стабильно функционировать при повторяющихся пользовательских действиях и при возникновении частных ошибок. Проведенная апробация показала, что приложение корректно обрабатывает авторизацию, разграничивает роли пользователей, поддерживает целостность структуры тестов и попыток, предотвращает повторное завершенное прохождение и сохраняет данные о результатах в базе. Дополнительно реализованы безопасные резервные сценарии на случай временной недоступности AI-сервиса. Благодаря этому система не теряет базовую функциональность даже при отказе внешнего интеллектуального компонента.

Удобство использования оценивалось с точки зрения двух целевых групп. Для преподавателя важными параметрами являются скорость создания теста, наглядность аналитики и доступность данных по студентам. Для студента — простота навигации, понятность процедуры прохождения теста и информативность итогового отчета. Реализованный интерфейс в целом удовлетворяет этим требованиям: он использует понятные визуальные паттерны, отображает ход выполнения теста, показывает детальный разбор результатов и предоставляет наглядные элементы сравнения с группой.

Отдельно следует подчеркнуть значимость визуальных механизмов обратной связи. Прогресс-бар во время прохождения теста, маркировка отвеченных вопросов, карточка итогового результата, сравнение со средним баллом по группе и отображение AI-комментариев повышают прозрачность работы системы. Пользователь не сталкивается с «черным ящиком», а получает достаточно подробное представление о собственных результатах. Для образовательной системы это особенно важно, поскольку доверие к оцениванию напрямую связано с понятностью механизма его формирования.

По совокупности рассмотренных критериев можно сделать вывод, что разработанная система демонстрирует достаточный уровень точности, надежности и удобства использования для уровня функционального прототипа. В дальнейшем эти показатели могут быть улучшены за счет расширения набора эталонных ответов, накопления размеченных данных, внедрения механизмов экспертной модерации AI-оценок и проведения более масштабного педагогического эксперимента.
